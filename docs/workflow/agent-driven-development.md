# Agent-Driven Development Rules

이 프로젝트는 **에이전트 드리븐**으로 개발한다. 학습/포트폴리오 자산이 무너지지 않도록 명확한 룰을 둔다.

## 핵심 원칙

> **타이핑은 위임 OK. 사고는 위임 NOT OK.**

면접관이 "이거 AI로 만든 거 아니에요?"라고 물으면 답은 **"네, 에이전트 드리븐으로 만들었습니다."** 가 되어야 한다. 단 모든 아키텍처 결정, 트레이드오프 평가, ADR 작성은 본인의 것이어야 한다.

## 위임 매트릭스

| 위임 OK (에이전트 활용) | 위임 NOT OK (본인이 직접) |
|---|---|
| 코드 타이핑 / 생성 | 아키텍처 결정 |
| 보일러플레이트 | 트레이드오프 평가 |
| 테스트 케이스 초안 | 모듈/인터페이스 경계 |
| 리팩토링 | 어려운 디버깅의 가설 수립 |
| 문서 어조 다듬기 | ADR 작성 (초안부터 본인) |
| 영문 번역 | 블로그 사고 흐름 |

## 매 PR 머지 전 단 하나의 셀프 체크

> **"이 코드, 6개월 뒤에 술자리에서 후배에게 1시간 설명할 수 있는가?"**

- YES → 머지
- NO → 에이전트에게 강한 질문 → 이해될 때까지 → 그제야 머지

이 체크 한 줄만 지키면 트랙 안 무너진다.

## 강한 질문 vs 약한 질문

ADR/블로그 quality는 질문 quality가 결정한다.

| 약한 질문 (성장 작음) | 강한 질문 (ADR/블로그 씨앗) |
|---|---|
| `sync.Map`이 뭐예요? | `sync.Map` vs `sync.Mutex + map`의 트레이드오프는? 우리 read/write 비율에선 어느 쪽이 적합? |
| 어떻게 해요? | 이 패턴의 대안 2-3개와 각각의 장단점은? |
| 맞아요? | 이 결정의 가정이 깨지는 시나리오는? |
| 버그 고쳐줘 | 이 버그의 근본 원인은? 비슷한 버그 재발 가능성은? 어떻게 방지? |
| 이거 쓰면 돼요? | 비슷한 일을 하는 라이브러리 비교 + 라이선스/유지보수 활동성/이슈 응답 시간은? |

매번 강한 질문 던지는 게 곧 ADR 작성 연습이다.

## ADR 작성 원칙

1. **비자명한 결정은 모두 ADR**
   - 라이브러리 선택
   - 패턴 vs 패턴
   - v0.1 포함 vs v0.2 연기
   - 인터페이스 모양

2. **포맷:** `Status` / `Context` / `Decision` / `Alternatives Considered` / `Consequences` / `Open Questions`

3. **에이전트 활용 허용 범위**
   - 영문 다듬기 ✅
   - 구조 검토 ✅
   - 비교 자료 리서치 ✅
   - **초안 통째로 생성 ❌** — 1차 초안은 본인이 직접 쓴다

4. **번호:** 0001부터 단조 증가, 폐기는 새 번호가 아닌 `Status: Superseded by ADR-NNNN`

## 블로그 작성 사이클

v0.1 6주 동안 최소 5편 출간. ADR 1개당 블로그 1편이 자연스러운 변환률.

| 시점 | 가제 |
|---|---|
| Week 2 | "Why I'm building a Go alternative to LiteLLM" |
| Week 3 | "Multi-provider failover patterns with goroutines" |
| Week 4 | "Distributed rate limiting for LLM APIs in Go" |
| Week 5 | "Observability for LLM workloads: what to measure" |
| Week 6 | "Agent-driven OSS development: my workflow & guardrails" |

각 글은 해당 시점의 ADR을 기반으로 확장. **ADR 깊이 = 블로그 깊이.**

## 안티패턴 (즉시 멈춰야 함)

- "테스트 통과하니까 머지"
- "AI가 좋다니까 머지"
- "PR description 안 읽고 머지"
- "ADR은 나중에 한꺼번에 쓰자"
- "영문 번역은 v1.0에 한꺼번에"
- "글은 v0.1 다 끝나고 한꺼번에"

→ 한 번 무너지면 트랙 무너진다. **매주 ADR 1편, 매주 또는 2주 블로그 1편** 페이스 유지.

## 작업 모드별 가이드

### "결정" 모드 (본인이 손으로)
- ADR 1차 초안
- 인터페이스 모양 잡기
- 라이브러리 선택
- 디렉토리/모듈 구조

### "수렴" 모드 (에이전트와 페어)
- 코드 작성 (line-by-line 확인 후 머지)
- 테스트 보완
- 문서 다듬기
- 리팩토링

### "검증" 모드 (본인이 손으로 다시)
- 머지 직전 셀프 체크 ("6개월 뒤 설명 가능한가")
- ADR 확정
- 블로그 출간

## 도구

- **금지:** 본 룰을 우회하는 모든 자동화 ("AI가 다 짜고 형은 머지만" 같은 흐름)
